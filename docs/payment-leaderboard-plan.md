# 功德榜支付宝接入与自动更新方案

## Summary

- 保留现有 `Cloudflare Pages + Next.js 静态导出` 主体，不改知识图谱和左侧实体排行榜的静态数据链路。
- 单独给功德榜增加动态链路：`Pages Functions + D1`，让支付成功后直接写库并实时刷新功德榜，不再依赖重新构建 `leaderboards.json`。
- 支付渠道只规划支付宝，不依赖 Stripe。由于当前没有已备案可用于支付的域名，方案按“两段式”设计：
  - 第一段先完成动态功德榜、订单流转、自动更新、敏感词过滤和支付适配层。
  - 第二段在支付宝应用资质和域名条件满足后接入正式支付宝下单与回调。
- 赞助用户支持自定义显示名字；榜一支持在最右侧留一句短话。
- 不做人工审核，改为服务端自动敏感词过滤，重点拦截涉政和政治敏感词；文本不合规时拒绝提交或降级展示。

## Current State

- 站点当前为纯静态导出，见 `site/next.config.mjs` 中的 `output: 'export'`。
- 左侧排行榜数据来自 `public/data/leaderboards.json`，由构建期静态读取。
- 右侧功德榜目前来自 `site/src/lib/constants.ts` 内的硬编码 `SPONSOR_RECORDS` / `SPONSORS_DATA`。
- 现有实现不具备：
  - 支付创建
  - 支付回调
  - 动态榜单聚合
  - 自定义展示名存储
  - 榜一留言存储
  - 自动文本过滤

## Architecture

- 保留现有页面路由与样式结构，只把功德榜数据源从硬编码改为动态接口。
- 新增动态能力：
  - `Cloudflare Pages Functions` 负责订单创建、支付回调、榜单查询。
  - `Cloudflare D1` 负责订单、用户展示信息和榜单聚合所需数据存储。
  - 支付适配层封装为 `alipay` provider，避免把支付宝签名、下单、回调逻辑散落在页面层。
- 自动更新机制：
  - 用户提交赞助表单。
  - 服务端创建订单并返回支付宝支付参数。
  - 支付宝异步通知成功后，服务端更新订单状态为 `paid`。
  - 功德榜接口按已支付订单实时聚合金额并返回最新排名。
  - 支付成功页轮询订单状态；页面刷新或轮询时即可看到更新后的榜单。
- 不写回 `web-data/leaderboards.json`，也不触发整站重建。

## Data Model

### `sponsor_orders`

- `id`: 本地订单 ID
- `order_no`: 外部展示订单号，唯一
- `provider`: 固定为 `alipay`
- `amount_fen`: 赞助金额，单位分
- `status`: `pending | paid | closed | failed`
- `currency`: 固定为 `CNY`
- `display_name_raw`: 用户提交的原始显示名
- `display_name_final`: 过滤后最终显示名；不合规时为空
- `is_anonymous`: 是否匿名
- `headline_raw`: 用户提交的榜一留言原文
- `headline_final`: 过滤后最终榜一留言；不合规时为空
- `headline_status`: `empty | approved | blocked`
- `text_filter_version`: 命中过滤规则时记录版本
- `alipay_trade_no`: 支付宝交易号
- `paid_at`: 支付成功时间
- `created_at`: 创建时间
- `updated_at`: 更新时间

### 排名口径

- 按“累计已支付金额”排序，不统计 `pending` / `failed` / `closed`。
- 同金额时，按最早达到该金额的支付时间排序。
- 同名用户不合并，榜单按订单维度累计到“显示身份”：
  - 匿名订单统一显示为 `匿名功德主`
  - 非匿名订单按 `display_name_final` 聚合
  - 若 `display_name_final` 为空，则显示为 `新晋功德主`

## Public Interfaces

### `POST /api/sponsor/orders`

- 入参：
  - `amountYuan`
  - `displayName`
  - `isAnonymous`
  - `headline`
- 服务端行为：
  - 校验金额范围和精度。
  - 对 `displayName`、`headline` 做长度限制和敏感词过滤。
  - 创建本地订单。
  - 调用支付宝下单适配层，返回支付跳转所需参数。
- 返回：
  - `orderNo`
  - `provider`
  - `payPayload`
  - `sanitizedPreview`

### `POST /api/payments/alipay/notify`

- 只接收支付宝异步通知。
- 服务端行为：
  - 校验签名。
  - 校验订单号、金额、交易状态。
  - 幂等更新订单状态为 `paid`。
  - 写入 `alipay_trade_no` 与 `paid_at`。
- 返回支付宝要求的成功响应文本。

### `GET /api/sponsor/orders/:orderNo`

- 用于支付成功页轮询订单状态。
- 返回：
  - `status`
  - `amountYuan`
  - `displayName`
  - `headline`

### `GET /api/sponsor/leaderboard`

- 返回动态功德榜数据。
- 每条记录包含：
  - `rank`
  - `displayName`
  - `amountYuan`
  - `title`
  - `headline`
  - `isAnonymous`
  - `updatedAt`

## Text Policy

### 长度限制

- 自定义显示名最多 `16` 个中文字符或等价长度。
- 榜一右侧小句话最多 `12` 个中文字符或等价长度。
- 超出长度直接拒绝提交，不做自动截断。

### 过滤规则

- 只做服务端校验，前端可做同样提示但不作为安全边界。
- 过滤对象：
  - `displayName`
  - `headline`
- 默认拦截范围：
  - 涉政和政治敏感词
  - 常见绕过写法：大小写变体、空格拆分、全半角混写、常见符号插入
- 实现策略：
  - 服务端先做标准化：去首尾空白、合并连续空格、全角转半角、统一大小写、去常见分隔符后再匹配。
  - 使用本地维护的敏感词 denylist 文件，首版直接随代码发布，后续可迁到 D1/KV。
  - 命中规则时返回明确错误，不创建可支付订单。
- 支付后兜底：
  - 若历史数据、绕过写法或脏数据导致文本未通过最终展示校验，则金额照常计入榜单，但显示名降级为 `新晋功德主`，榜一留言不展示。

## Frontend Changes

- 保持当前功德榜视觉风格和像素排序图标不变。
- 将 `SPONSORS_DATA` 替换为从动态接口读取的数据。
- 新增赞助表单区域：
  - 输入金额
  - 自定义显示名
  - 匿名开关
  - 榜一留言输入框
  - 支付按钮
- 交互规则：
  - 只有当前支付金额足以成为榜一时，才允许填写并提交榜一留言；否则输入框禁用并给出说明。
  - 若支付后因其他人抢先成为榜一，则该订单留言不展示，但数据保留。
  - 移动端默认不展示榜一右侧短句，避免压坏现有表格布局。
  - 桌面端榜一短句单行显示，超出容器时省略号截断。

## Payment Integration

### 第一阶段

- 完成动态榜单、订单模型、接口、文本过滤和前端交互。
- 完成 `alipay` 适配层接口定义：
  - `createOrder()`
  - `verifyNotify()`
- 在没有正式支付宝资质可联调前，保留 provider stub 或沙箱适配逻辑。

### 第二阶段

- 接入支付宝正式应用。
- 需要具备：
  - 企业主体支付宝开放平台应用
  - 已签约支付产品
  - RSA2 密钥对
  - 可用回调地址
  - 至少一个满足支付宝要求的正式域名/子域名
- 若支付宝网页支付对备案域名有强校验，则支付页与回调接口迁移到已备案域名；主站榜单仍可保留现有域名并跨域读取接口。

## Acceptance Criteria

- 用户支付成功后，功德榜在不重建站点的前提下自动更新。
- 用户可自定义显示名，且合规文本会自动展示。
- 榜一用户可在桌面端最右侧展示一句短话，不破坏现有布局。
- 不需要人工审核；涉政/政治敏感词会在服务端被拦截。
- 重复支付回调不会重复记账。
- 现有左侧知识图谱排行榜和静态页面构建流程不受影响。

## Test Plan

- 订单创建
  - 合法金额可正常创建。
  - 非法金额、超长显示名、超长留言被拒绝。
  - 命中政治敏感词的显示名和留言被拒绝。
- 文本过滤
  - 全半角混写、空格拆分、大小写变体、符号插入的敏感词仍能被拦截。
  - 普通名字和常见品牌名不会被误杀。
- 支付回调
  - 签名错误拒绝。
  - 金额不一致拒绝。
  - 重复通知只记账一次。
- 榜单聚合
  - 已支付金额正确累加。
  - 同金额排序稳定。
  - 匿名、空展示名、被降级展示名都按预期显示。
- 前端展示
  - 桌面端榜一短句不撑破表格。
  - 移动端不显示榜一短句时布局正常。
  - 支付成功页轮询后能看到最新榜单。

## Assumptions

- 默认部署方案为 `Cloudflare Pages + Functions + D1`，不新增传统服务器。
- 默认金额单位为人民币元，后端统一换算为分。
- 默认显示名上限为 `16` 字，榜一短句上限为 `12` 字。
- 默认敏感词策略以“政治敏感词拦截”为首要目标，词库先随代码维护。
- 默认不做退款回滚榜单；后续若需要，再补退款同步逻辑。
- 默认不把支付数据并回 `web-data/leaderboards.json`。
